# 業績データベース　運用マニュアル（管理者むけ）

> 対象：研究室の管理者（教員・スタッフ）。日々の確認・反映を担当する人。
> このマニュアルは **日々の運用** に絞っています。
> フォームやデータベースの**初期セットアップ**（作成手順・移行）は別資料を参照してください
> → [`docs/google-forms.md`](google-forms.md) ／ [`docs/migration-guide.md`](migration-guide.md)

---

## 1. あなたの役割（全体像）

メンバーがフォームから送った業績は、**「未確認」の状態**でデータベース（Canonical スプレッドシート）に溜まります。
あなたの仕事は、それを**確認して「確認済」にする**ことです。**「確認済」にした行だけ**がアプリに表示されます（品質ゲート）。

```
メンバーが送信 → DB に「未確認」で溜まる
   ↓（あなた）
内容を確認 → 「確認済」にする
   ↓
業績まとめアプリに反映される
```

- 日々やることは、ほぼ **「点検 → 承認」の 2 つだけ**です。
- 操作はスプレッドシート上部の **「業績DB」メニュー** から行います（クリックだけ）。

---

## 2. 毎回の運用フロー（これだけ覚えれば OK）

### ステップ 1　★ まとめて点検
スプレッドシートを開き、上部メニュー **「業績DB」→「★ まとめて点検（重複・欠損・色分け → レポート）」** を押します。

これ 1 つで、全タブに対して次が自動実行されます。
- **重複の判定**：同じ DOI／同じ日付＋タイトルなどを検出し、`note` 列に `dup_of=<既存ID>` を付ける。
- **欠損の判定**：種別ごとの必須項目が欠けていれば `note` 列に `missing=<項目>` を付ける。
- **色分け**：欠損＝**橙**／未確認＝**黄**／重複候補＝**赤の太字**。
- **`点検レポート` シートを生成**：問題のある行だけを一覧化（タブ／ID／種類／詳細／タイトル）。

> → まずは **`点検レポート` シートを上から見て対処**すれば OK。問題がなければ「問題は見つかりませんでした」と出ます。

### ステップ 2　問題のある行を直す
`点検レポート` を見ながら対応します（詳しくは「4. 重複・欠損への対処」）。
- **重複**：どれと重複しているか（一致先 ID とタイトル）が並びます。不要な行を削除するか、`未確認` のまま保留。
- **欠損**：足りない項目が分かります。可能なら補う（自動補完は「3. 自動補完」）。

### ステップ 3　承認（未確認 → 確認済）
内容に問題がなければ、対象の行を選択して
**「業績DB」→「選択行を承認(確認済)」** を押します。
- 複数行をまとめて選択して一括承認できます。
- これが**アプリへの反映条件**です（`status` を手で `確認済` に書き換えても同じ）。
- 戻したいときは **「選択行を未確認に戻す」**。

> **※** 著者名の表記ゆれ（`Hayashi N` / `林 直子` など）は**気にしなくて構いません**。アプリ側が名簿をもとに自動で名寄せします。

---

## 3. 自動補完（手入力を減らす）

足りない情報を、外部データベースから自動で補えます。

- フォーム送信時にも自動で動きますが、あとから手動でも実行できます。
- 対象行を選択して **「業績DB」→「選択行を補完（DOI・タイトル・ISBN 自動）」**。

| 種別 | 補完のしくみ |
|---|---|
| **論文** | DOI があれば CrossRef で英語情報を補完。DOI が無くてもタイトル（＋著者）から DOI を特定して補完。 |
| **書籍** | DOI で CrossRef、ISBN で OpenLibrary（書名・出版社・出版日）を補完。 |

- 補完されると `note` 列に補完元（`crossref` / `crossref-title` / `openlibrary`）が記録されます。
- **空欄だけ**を埋めます（既に入っている値は上書きしません）。
- タイトルからの DOI 特定は、誤った文献を拾わないよう一致度で検証するため、**特定できないこともあります**（その場合は手入力）。

---

## 4. 重複・欠損への対処

### 重複（`note` に `dup_of=...`）
- 同じ業績が二重登録されている目印です。`dup_of=<既存ID>` の ID が「重複先」です。
- 対応：**新しい方を削除**するか、判断に迷う間は `未確認` のままにしておきます。
- メンバーが同じものを 2 回送ったときによく出ます。消しても消さなくてもアプリの表示が壊れることはありません。

### 欠損（`note` に `missing=...`）
- 必須項目が空の目印です。`missing=` のあとに足りない項目名が出ます。
- 対応：可能なら補完（「3. 自動補完」）や手入力で埋め、無理なら報告者に確認。
- 埋めたあと、もう一度 **「★ まとめて点検」** を押すと最新化されます。

---

## 5. メンバーが自分で修正できるようにする（任意）

フォームは送信後に本人が直せません。メンバーに DB の編集権を渡すと、本人が後から修正できます（安全策つき）。

- 一度だけ「業績DB →メンバー編集を有効化」を実行しておくと、
  - 列名・メタ列（`status` など）が**保護**され、メンバーは触れません。
  - メンバーがデータを直すと、その行が自動で **`未確認` に戻り**、`note` に `edited` が付きます。
- つまり **メンバーの修正も必ず再確認を通ります**。`edited` が付いた行＝再確認の対象です。

> **注意：** 共有はシート単位のため、技術的には他人の行も編集できてしまいます。**少人数・信頼できる範囲での運用**を前提にし、メンバーには「自分の行だけ直す」と周知してください。厳格に分けたい場合は、編集権を渡さず**修正は管理者が行う**運用にします。

---

## 6. 名簿（著者の名寄せ）について

- 著者名の表記ゆれは、**名簿シート（`Input confirmation`）** の和名↔ローマ字をもとに自動で吸収されます。
- 新しいメンバーが増えたら、名簿に **和名とローマ字表記** を追記してください。これで、その人の業績が正しく集計・強調されます。
- 名簿に無い人でも、あいまい一致（rapidfuzz）である程度は吸収されますが、**正確に集計したい人は名簿に載せる**のが確実です。

## 7. 業績まとめアプリの使い方

確認済みの業績を、提出先に合わせて整形・コピーするアプリです。

### 起動・データの指定
- アプリを開き、サイドバーの **データソース** に対象スプレッドシートの URL または ID を入れます（既定値あり）。
- データを最新化したいときは、サイドバーの **「再読み込み」**（円形矢印アイコン）を押します（フォーム承認の反映もこれで）。

### 絞り込み（サイドバー）
- **業績種別**：表示する種別を選ぶ（複数可）。
- **著者**：選ぶと、その人の業績だけに絞り込み、**出力中のその名前を自動で太字**にします。
- **年度**：4 月始まりの年度で範囲指定。
- **査読ありのみ**：論文・著書にだけ作用（発表・受賞などは消えません）。
- **国内/国際**：発表・受賞・アウトリーチにだけ作用。
- **照合のゆるさ**：著者名寄せの厳しさ。通常は既定（85）のままで OK。

### 書式の調整
- **プリセット**（科研費 など）を選ぶだけで整形されます。
- 細かく直したいときは各セクションの **「書式を調整する」** を開き、`{ }` で囲んだ項目の意味を見ながら編集できます。
- **太字・斜体**は項目を選ぶだけ。**著者の表示人数**（0＝全員）や省略語（「ほか」「et al.」）も種別ごとに設定できます。

### 出力（コピー）
- **リッチ表示をドラッグして選択 → コピー**すると、太字・斜体ごと Word などに貼り付けられます。
- プレーンテキストが必要なときは「プレーンテキスト」の折りたたみを開きます。
- 複数種別を表示中は、末尾の **「すべてまとめてコピー」** で全種別を 1 つにまとめてコピーできます。
- **該当 0 件の種別・全 0 件の著者は自動で非表示**になります。

---

## 8. 限定公開・セキュリティ（運用上の注意）

- アプリは「閲覧のみ」公開で運用できます。学外に出したくない場合は、パスワード保護や Streamlit Cloud の Private 設定が使えます（詳細は [`README.md`](../README.md)）。
- DB（スプレッドシート）の**編集権は最小限**にしてください。アプリ用に「閲覧」を開ける場合も、原則として編集権は付けません（例外：「メンバー編集を有効化」で本人修正を許す場合のみ、信頼できる範囲に付与）。

---

## 9. 困ったとき（運用チェックリスト）

| 症状 | 確認すること |
|---|---|
| メンバーの業績がアプリに出ない | その行の `status` が `確認済` になっているか。なっていなければ承認する。 |
| 反映されたはずなのに表示されない | アプリのサイドバー **「再読み込み」**（円形矢印アイコン）を押したか。 |
| 同じ業績が二重に出る | `note` の `dup_of` を確認し、不要な行を削除。 |
| 著者でうまく絞り込めない | 名簿（`Input confirmation`）にその人の和名・ローマ字があるか。 |
| 必須が空の行がある | 「★ まとめて点検」の `missing` を確認し、補完または報告者へ確認。 |

> 入力者（学生）むけの操作は、別資料「入力者むけマニュアル」を参照・配布してください。
