# 業績入力フォーム（1 フォーム＋自動取込）作成手順

各メンバーが **1 つの Google フォーム** で業績を送信 → 回答が **自動で** 業績 DB（Canonical スプレッドシート）の該当タブへ蓄積 → 本アプリ（研究業績サマライザー）で抽出・整形する運用にするためのガイド。

> 以前は「種別ごとに 6 フォーム」を作る設計でしたが、**リンクが 6 本に分かれて使いにくい／作成・保守が重い／回答を DB へ手で転記する手間が残る** という問題がありました。
> 本手順は **フォーム 1 つ・配布リンク 1 本** に統合し、送信から DB 追記までを Apps Script で自動化します。

---

## 全体像

```
メンバー   1 本のリンクを開く → 先頭で「業績種別」を選ぶ → その種別の項目だけ入力 → 送信
   │
   ▼ （Apps Script: onFormSubmit トリガが自動起動）
業績 DB    Canonical スプレッドシートの該当タブへ 1 行追記（status=未確認, source=form）
   │
   ▼ （あなた＝キュレーターが内容を確認）
反映       status を「確認済」に変えた行だけ、アプリに表示される（品質ゲート）
```

- **メンバーの手間**：リンク 1 本。種別を選ぶと、その種別に必要な欄だけが出る。
- **あなたの手間**：フォーム作成はスクリプトを 1 回実行するだけ。日々は「未確認 → 確認済」を確認するだけ
  （後述の**キュレーター・メニュー**でワンクリック一括承認できる）。
- **持続可能性**：フォームの定義はスクリプト（`scripts/forms/publication_form.gs`）にあり、項目を変えたら再生成できる。手作業のセクション組みや回答の手転記は不要。

### 自動でやってくれること（送信時）
- **英日両方入力**：タイトル・雑誌名/学会名などは日本語と英語の両欄を用意。両方入れればアプリの表示言語で自動選択、片方だけなら有る方を使う。
- **DOI 自動補完**：論文・著書で DOI を入れて送ると、CrossRef を参照して英語タイトル・雑誌名・巻号ページ・発行日の**空欄だけ**を補完する（`note` に `crossref`）。
- **タイトルから DOI 検索（論文のみ）**：論文で **DOI 未入力**でも、タイトル（＋著者）で CrossRef を検索し、**十分一致する候補が見つかれば** DOI を自動で特定し、上記の DOI 補完を実行する（`note` に `crossref-title`）。誤った文献を拾わないようタイトルの一致度で検証するため、特定できないこともある（その場合は手入力 or 後述のメニューで再試行）。
- **重複フラグ**：同じ DOI、または同じ日付＋タイトルが既にあると、`note` に `dup_of=<既存ID>` を付けて**残す**（消さない）。あなたが見て削除・統合を判断。誤検出（実は別物）の場合は、キュレーター・メニューの「選択行を『重複ではない』に確定」で `not_dup=<相手ID>` に変え、以後その相手とは再フラグされないようにできる（[`admin-guide.md`](admin-guide.md) 参照）。

> ⚠️ これらを有効にするには、DB が **v2 スキーマ**（`_ja`/`_en` 列＋`note` 列）である必要があります。
> 既存 DB は [`migration-guide.md`](migration-guide.md) の「DB を v2 へ上げる」を一度だけ実施してください。

### メンバー自身が「まとめて貼り付け」で一括登録（CLI 不要）
種別選択の最後にある **「まとめて貼り付け（複数件を一度に）」** を選ぶと、貼り付け用セクションへ進みます。
メンバーは **researchmap 等からコピーしたプレーンテキスト**を貼って送信するだけで、複数件が一度に登録されます。

- 「貼り付ける業績の種別」を 1 つ選ぶ（1 回の送信につき 1 種別。論文と発表は分けて送信）。
- 「業績リスト」に貼り付け。1 件の並びは **タイトル → 著者 → 誌名/学会名＋日付**。
  **末尾に日付がある行（例: `2025年7月` / `Jul, 2025` / `2025/07`）が 1 件の区切り**です。
- 送信すると Apps Script が解析し、巻号・ページ・日付を自動抽出、タイトル/誌名/学会名は本文の言語で
  `_ja`/`_en` に自動振り分けて、各件を `status=未確認` で追記（重複は `dup_of` フラグ）。
- 解析の都合上、**1 項目 1 行・余計な注記なし**で貼ってください（researchmap のコピーそのまま）。

#### 一括貼り付けを LLM で自動構造化する（任意・精度向上）

トークンを 1 つ登録しておくと、**「まとめて貼り付け」送信時に自動で GitHub Models（OpenAI 互換・無料枠）が解析**し、崩れた表記でも取りこぼしを減らせます。設定しなければ従来のヒューリスティック解析のまま動きます（後方互換）。

1. GitHub で **Personal Access Token（fine-grained）** を発行：Settings → Developer settings → Fine-grained tokens → **Account permissions の「Models」を Read-only** にして生成。
2. フォーム用の Apps Script プロジェクトを開き、**⚙ プロジェクトの設定 → スクリプト プロパティ** で、キー `GITHUB_MODELS_TOKEN`・値に発行した PAT を追加して保存。
3. 以降、メンバーの「まとめて貼り付け」送信は自動で LLM 構造化される（`source` 列が `paste-llm`）。

- **DOI は LLM に生成させません**（本文に明記がある時だけ採用）。DOI/英語情報の確定は従来どおり CrossRef が補完します。
- **失敗時（トークン無・API/JSON エラー）は従来解析へ自動フォールバック**するので、送信が失敗することはありません。
- レート制限はトークン所有者（あなた）の無料枠を消費します（目安: 150 req/日）。1 送信＝1 リクエストです。
- **プライバシー**：貼り付け本文が GitHub Models（外部）へ送信されます。`source=paste-llm` の行は LLM 解析由来、`paste-form` は従来解析由来として見分けられます。

> CLI（`scripts/ingest_paste.py --llm`）でも同じ LLM 構造化ができます（`GITHUB_TOKEN` を環境変数に設定）。**メンバー配布にはこのフォームの貼り付けが最も簡単**です。

#### CLI で精度を上げる：LLM 構造化（`--llm`、GitHub Models）

貼り付けテキストの解析は通常ヒューリスティック（末尾の日付で区切る等）ですが、researchmap 以外の崩れた表記では取りこぼしが出ます。**`--llm` を付けると GitHub Models（OpenAI 互換・無料枠）でテキストを構造化抽出**し、手直しを減らせます。

```powershell
# 1) GitHub の Personal Access Token (fine-grained) を models:read 権限で発行し、環境変数に設定
$env:GITHUB_TOKEN = "github_pat_xxx"
# 2) --llm を付けて取り込み（失敗時は自動で従来解析にフォールバック）
.\.venv\Scripts\python.exe scripts\ingest_paste.py --type paper --src papers.txt --llm --append canonical.xlsx
#    別モデルを使う場合: --model openai/gpt-4o
```

- **DOI は LLM に作らせません**（捏造防止）。DOI は本文に書かれている時だけ拾い、確定・補完は CrossRef（フォーム送信側の自動補完／キュレーターメニュー）に委ねます。
- **トークン未設定・`openai` 未導入・API/JSON エラー時は、警告のうえ従来のヒューリスティック解析へ自動フォールバック**します（`--llm` を付けても失敗で止まりません）。`openai` は `requirements.txt` の任意依存です。
- **無料枠にレート制限**があります（low-tier の目安: 15 req/分・150 req/日・入力 ~8000 トークン）。本ツールは 1 回の貼り付けを 1 リクエストにまとめ、長文は自動でチャンク分割します。
- **プライバシー注意**：`--llm` を使うと貼り付け本文が GitHub Models（外部サービス）へ送信されます。機微な情報を含む場合は使用可否を確認してください。将来ローカル LLM へ切り替える場合も、`--model` と接続先（OpenAI 互換）の差し替えだけで対応できる設計です。

> **設計方針（DOI を必須にしない）**：論文以外（発表・受賞・著書・アウトリーチ・広報）に DOI は無く、和文誌・和書は DOI 解決が弱いため、全項目を手入力で受けます。DOI は論文・著書の **任意項目**（形式検証つき）に留めます。

## 準備するもの

- 業績 DB となる **Canonical スプレッドシート**（種別ごとのタブを持つ。作り方は [`migration-guide.md`](migration-guide.md) を参照）。その **スプレッドシート ID** を控えておく（URL の `/d/` と `/edit` の間の長い文字列）。
- 上記スプレッドシートを **編集できる Google アカウント**（＝あなた）。スクリプトはこの権限で DB に書き込みます。アプリ側の「閲覧のみ公開」はそのままで構いません。

## 作成手順（1 回だけ）

### 1. Apps Script プロジェクトを作る
1. <https://script.google.com> を開き、右上が業績管理用の Google アカウントか確認。
2. **「新しいプロジェクト」** をクリック。
3. 既定の `コード.gs` の中身を**全部消す**。

### 2. スクリプトを貼り付ける
1. リポジトリの **`scripts/forms/publication_form.gs`** を開き、**全文をコピー**。
2. 手順 1 で空にしたエディタへ**貼り付け**。
3. 上部の保存アイコン（💾）で保存。プロジェクト名は「業績登録フォーム」等に。

### 3. 書き込み先 DB を指定する
貼り付けたコードの先頭付近にある次の 1 行を探し、ダブルクォートの中身を**自分の Canonical スプレッドシート ID** に書き換えて保存：

```javascript
var CANONICAL_SHEET_ID = "1_7b_-6EsRNr6tW1naDj8QJ5M0espE5Wv";
```

> ここが**唯一**書き換える箇所です。種別ごとの設問は `FIELD_MAP` に定義済みで、`schema.py` と一致しています（`tests/test_form_fields.py` が自動チェック）。

### 4. フォームを生成する（認可）
1. エディタ上部の関数選択で **`buildForm`** を選ぶ。
2. **「実行」** をクリック。
3. 初回は認可ダイアログが出ます → **自分のアカウントを選択** →「詳細」→「（プロジェクト名）に移動」→ **許可**。
   （Google フォームの作成・スプレッドシートへの書き込み・トリガ設置を許可します。）
4. 実行が終わると、下部の**実行ログ**に次が出ます：
   ```
   公開用 URL（メンバーへ配布）: https://docs.google.com/forms/d/e/.../viewform
   編集用 URL（あなた専用）:   https://docs.google.com/forms/d/.../edit
   ```
   ログが見えないときはメニュー **表示 → ログ**（または `Ctrl+Enter`）。

> ⚠️ `buildForm` を**もう一度実行すると新しいフォームが別に作られます**。やり直したいとき以外は再実行しないでください（設問を増減したいときは `FIELD_MAP` を直してから作り直す運用）。

### 5. 動作をテスト送信で確認する
1. **公開用 URL** を開き、種別を 1 つ選んで 1 件テスト送信。
2. Canonical スプレッドシートの該当タブ（例：`Original Papers`）を開き、最下行に
   `status=未確認`・`source=form`・`record_id=PAP-xxxx` の行が増えていれば成功。
3. その行の `status` を **`確認済`** に変更 → アプリを開き、件数に反映されることを確認。
4. 確認できたらテスト行は削除して構いません。

### 6. メンバーへ配布する
**公開用 URL** をメール・チャット等で共有します。リンクはこの 1 本だけです。
フォームの説明欄（生成時に設定済み）に、記入例・締切・問い合わせ先を追記しておくと親切です。

## 確認をラクにする：キュレーター・メニュー（`canonical_tools.gs`）

確認作業（未確認→確認済、DOI 補完、重複チェック）をスプレッドシート上のメニューから行えます。

1. **Canonical スプレッドシート**を開く →「拡張機能」→「**Apps Script**」。
2. `scripts/forms/canonical_tools.gs` を全文コピペして保存。
3. スプレッドシートを再読み込みすると、上部に **「業績DB」メニュー**が出る（初回操作時に認可）。

メニュー項目（record タブで行を選択してから実行）:
- **選択行を承認(確認済)** … 選んだ行の `status` を一括で `確認済` に（＝アプリに反映）。
- **選択行を未確認に戻す** … `status` を `未確認` に。
- **★ まとめて点検（重複・欠損・色分け → レポート）** … これ 1 つで全タブを点検する（下記「まとめて点検」）。日々の確認はこれだけでOK。
- **選択行を補完（DOI・タイトル・ISBN 自動）** … 選んだ行をタブと内容に応じて自動補完。**論文**＝DOI があれば CrossRef、無ければタイトル（＋著者）で DOI を特定してから CrossRef。**書籍**＝DOI で CrossRef ＋ ISBN で OpenLibrary（書名・出版社・出版日）。`note` に補完元（`crossref`/`crossref-title`/`openlibrary`）が付く。フォーム送信時にも DOI・ISBN があれば自動補完される。
- **メンバー編集を有効化（保護＋編集で未確認へ）** … メンバーが**自分の入力を後から修正**できるようにする（下記「メンバーが自分で修正できるように」）。

### まとめて点検（チェック者の手間を最小化）

**「業績DB → ★ まとめて点検」** を押すだけで、全 record タブに対して次を一括実行します。

1. **重複の再判定**（同一 DOI / 同一日付＋タイトル / 論文・著書は同一タイトル）→ 後発行に `note=dup_of=<既存ID>`。
2. **欠損の再判定**（種別ごとの必須項目。フォームの必須設問と整合）→ 欠けていれば `note=missing=<項目|...>`。
3. **色分け**：**欠損＝橙背景** / **未確認＝黄背景** / **重複候補＝赤の太字**。
4. **`点検レポート` シート**を生成し、**問題のある行だけ**を一覧化（タブ / `record_id` / 種類 / 詳細〔重複の一致先 or 欠損項目〕/ タイトル）。

→ チェック者は **`点検レポート` シートを上から見て対処**するだけです。重複は「どれとどれか（一致先 ID とタイトル）」がそのまま並びます。欠損は何の項目が足りないかが分かります。対処後にもう一度押せば最新化されます。問題が無ければ「問題は見つかりませんでした」と出ます。

### メンバーが自分で修正できるように（任意・送信後の見直し対策）

フォームは送信後に回答を見直せないため、**入力ミスを後から直せない**のが弱点でした。これを補うため、メンバーに DB の編集権を渡しても安全になる仕組みを用意しています。

1. 「業績DB →**メンバー編集を有効化**」を 1 回実行（初回は認可ダイアログで「許可」）。
   - 全 record タブの **ヘッダ行（列名）とメタ列**（`record_id`/`status`/`submitter`/`source`/`created_at`/`note`）を**保護**（編集はあなただけ）。
   - **編集トリガ**を設置：メンバーがデータ列を直すと、その行の `status` を自動で **`未確認` に戻し**、`note` に `edited` を付ける。
2. Canonical スプレッドシートをメンバーに **編集可（リンクを知っている人 or 指定アカウント）** で共有する。
3. メンバーは自分の行（`submitter` が自分の行）を見つけ、**データ列だけ**を修正。列名や `status` は保護されていて触れない。
4. 修正された行は `未確認` に戻るので、あなたが内容を確認して `確認済` に戻す（＝**編集も必ず最終チェックを通る**）。

> 💡 **チェック担当を複数名にする場合**：保護範囲（ヘッダ・メタ列）と承認操作は、初期状態ではオーナー 1 名しか行えません。2 名以上で運用するときは、`scripts/forms/canonical_tools.gs` の `ADMIN_EDITORS` に担当者のメールアドレスを追記してから「メンバー編集を有効化」を再実行します（手順は下記）。

> ⚠️ **割り切り**：共有は「シート単位」のため、メンバーは技術的には**他人の行の編集や行削除**もできてしまいます（行単位の保護は運用が非現実的）。少人数・信頼できる範囲での運用を前提に、メンバーには「**閲覧と、自分の行のみ修正**」を周知してください。厳格に分離したい場合は、編集権を渡さず**問い合わせ先**を案内し、修正はあなたが行う運用にします。
>
> アプリ用に「閲覧のみ」公開しているリンクとは別に、**編集権はこの用途に限って**付与してください。

### チェック担当を 2 名以上にする手順

初期状態では、保護されたヘッダ・メタ列の編集や承認操作は**オーナー 1 名だけ**ができます。下記の手順で担当者を追加すると、その人も承認・点検・メタ列修正ができるようになります。

1. **その人をシートに「編集者」で共有する**
   - スプレッドシート右上「共有」→ 担当者のメールを追加 → 役割「編集者」。
   - ※ 保護範囲の編集者にするには、先にシート自体の編集権が必要です。

2. **スクリプトに担当者のメールを登録する**
   - 「拡張機能 → Apps Script」を開き、`canonical_tools.gs` の `ADMIN_EDITORS` を編集します。

   ```javascript
   var ADMIN_EDITORS = ["checker@example.com"];          // 1 名追加
   // var ADMIN_EDITORS = ["a@example.com", "b@example.com"];  // 複数名はカンマ区切り
   ```

   - 保存（フロッピーアイコン or `Ctrl+S`）。

3. **「メンバー編集を有効化」を再実行する**
   - スプレッドシートに戻り、メニュー **「業績DB → メンバー編集を有効化（保護＋編集で未確認へ）」** を実行。
   - 既存の保護が作り直され、登録した担当者が保護範囲の編集者として追加されます。

4. **担当者側で 1 回だけ認可する**
   - 追加した担当者が初めてメニュー（承認・点検など）を使うときに認可ダイアログが出るので「許可」します。これで承認・点検が使えるようになります。

> ✅ 確認方法：担当者のアカウントで `status` 列を手動で書き換えてみて、保存できれば成功です（オーナー以外でも保護列を編集できる＝登録済み）。
> 担当者を外したいときは `ADMIN_EDITORS` から削除して、もう一度「メンバー編集を有効化」を実行します。

## 日々の運用（未確認 → 確認済）

- フォーム送信は **`status=未確認`** で DB に積まれ、アプリには出ません（下書き扱い）。
- 内容を確認したら、**「業績DB」メニュー → 選択行を承認(確認済)** でまとめて承認（または `status` を手で `確認済` に）。これだけがアプリへの反映条件です（品質ゲート）。
- **著者名の表記ゆれ**は気にしなくて構いません。名簿（`Input confirmation`）との名寄せをアプリ側（`roster.py`）が吸収します。
- **重複**は `note` の `dup_of=...` が目印。確認時に行を削除するか `未確認` のままにしてください。
- **メンバー編集を有効化**している場合、`確認済` の行をメンバーが直すと自動で `未確認` に戻り（`note` に `edited`）、再度の承認が必要になります。`edited` 付き＝メンバー修正後の再確認対象、と覚えておくと便利です。

## 限定公開（学外に出さない）

- フォーム設定（⚙）→「**回答**」→「**回答を 1 回に制限する／組織内に限定**」を必要に応じて ON（同一 Google Workspace 組織のみ・ログイン必須にできる）。
- 個人 Google アカウント運用で組織制限が使えない場合は、**リンクを限られた範囲にのみ共有**して運用します。
- 回答が積まれる Canonical スプレッドシート自体の共有範囲も最小化してください（アプリ用に「閲覧」を開ける場合も、原則として編集権は付けない）。
  - 例外として、メンバーに送信後の自己修正を許す場合は「**メンバー編集を有効化**」（前掲）で列名・メタ列を保護したうえで、**信頼できる範囲にのみ**編集権を付与します。

## 設問を増減したいとき

1. `scripts/forms/publication_form.gs` の `FIELD_MAP` を編集（種別・設問・必須・選択肢）。
   - フィールドを増減する場合は、`schema.py`／`ingest_to_canonical.py`／Canonical のヘッダも合わせて更新します。
   - `tests/test_form_fields.py`（`tests/verify.py` からも実行）で一致を検証できます。
2. `buildForm` を実行して**新しいフォームを作り直し**、新しい公開 URL を配布。
3. 旧フォームは受付終了（設定 →「回答」→「**回答を受付中**」を OFF）にして凍結。

## 補足：このスクリプトがしていること

- `buildForm()`：1 つのフォームを生成し、先頭に「報告者氏名」「業績種別（ラジオ）」を置き、種別の選択でその種別のセクションへ分岐させる。各設問とフィールドの対応（item ID → 種別・論理フィールド）を保存し、`onFormSubmit` トリガを設置する。
- `route(e)`：送信時に起動。選ばれた種別の値だけを集め、Canonical の該当タブのヘッダ順に 1 行追記する（`record_id` 採番・`status=未確認`・`submitter`＝報告者氏名・`source=form`・`created_at`）。タブ名・ヘッダ・`status` 文言は `loader.py`／`schema.py` と一致。
